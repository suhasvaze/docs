# 1. Chart of Accounts

Chart of Accounts is List of Accounts as per Traditional Terminology

# 1. What is COA

COA = Chart of Accounts

  1. In Traditional Accounting, COA means just the List of Accounts. And there could be Multiple Parameters on which Accounting Data can be analysed (e.g. by Project, by Customer, by Supplier, by Employee, by Legal Entity etc) E.g. View Sample Traditional COA

  2. In Oracle, COA can be made up of Multiple Segments. Some Analytical Aspects can be intertwined together as COA Segments for better Analytics and Data Capture. List of Accounts becomes just one segment in this scheme. View Sample Oracle COA




















# 2. Control, Analysis and Reporting

Different companies would have different Control/Analysis/Reporting requirements with-respect-to COA

  1. Analytical parameters can be unique to every company. E.g. Cost Centers, Products, Prod. Categories, Depts/Divisions, Business Units, Legal Entities, Customers, Suppliers, Employees etc

  2. Scenario-1 A Company dealing with only 4 large customers may want to track Expenses/Income by Customer. i.e. Requiring a P&L by Customer.

  3. Scenario-2 A Retail Company would want to draw P&L Ac by each Store/Shop. Per Sqft Profitability can help space allocation decisions.

  4. Scenario-3 A Software Services company would require to prepare P&L Ac by Project, and by Technology.

  5. Controlling Expenses by Departments would be a common practice in many companies.

  6. A Product Based company would analyse cost build-up of Products.

  7. A Lens Manufacturing Company would require Profitability by each Laboratory

  8. An Aluminum Rolling Company would require Profitability and Control by each Plant.

Multi-Segmented COA Structure can fulfill such requirements by re-using a common set of features. E.g. Features programmed for one segment can be used for all segments as needed




















# 3. Multi-Segmented COA

  1. As discussed earlier, Every Company would require a unique selection of dimensions (i.e. Products, Product Categories, Regions, Departments/Divisions, Business Units, Employees, Large Customers, Large Suppliers, Projects, Plants, Labs etc) for Reporting/Analysis and Control.

  2. To address such vast diversity of Requirements, Oracle allows you to flexibly Configure a Chart of Accounts with Multiple Dimensions.

E.g. After detailed study, A Company may select following segments to be embedded with COA

Company - To allow company wise Balance Sheet/P&L and Intercompany Trx

Departments - To exercise Department wise Cost Control

Account - This is mandatory Dimension (Traditional List of Accounts)

Sub Account - Flexibly Breakup Accounts as required in case of each account

Product - For Product wise Cost Collection and Profitability




















# 4. Benefits of Multiple Segments

  1. Data Relating to Each Dimension is entered across ALL TRANSACTIONS of ALL SUBLEDGERS. E.g. (AR) Receivables Invoices, (AP) Payables Invoices, (INV) Material Receipts, (FA) Asset Addition etc

  2. Budgeting, Encumbrance and Actual Tracking of Costs/Expenses for selected dimensions

  3. Easier Processes e.g. Consolidation process, Period Close etc.

  4. Common standards across All Regions, Group Companies and Related Entities

  5. Re-usable common set of features




















# 5. Single-point COA Control

  1. As a common Business Practice, the Chart of Accounts will be maintained by a "Single Group of Authorized Users" (You may call it "COA Maintenance Group"). Other users can only select the COA Values for transactions.

  2. If a New COA Value is required, the Users can Send the Request to the "COA Maintenance Group" and then the "COA Maintenance Group" will decide the Code to be used and Define the Value as part of the COA

  3. This process will ensure streamlined COA and eliminate duplicates and incorrect definitions of COA Values




















# 6. COA Setup Components

How to Create several List of Values (Dimensions) and then combine all these dimensions into one Chart of Accounts (called as Accounting Flex Field)

















# 1. Value Sets

Task: Manage Chart of Accounts Value Sets

  1. A Value Set means a code/name given to a List. They are used across Fusion, not just for COA.
  2. Above Task will allow you to Create Value Sets (and to View Value Sets already created)
  3. E.g. Code and Name of List of Companies may be Code: OEG_CO Name: Company
  4. Likewise, if you have 4 Lists (Co, A/c, CC, Product), you will need 4 Value Sets
  5. Values should not be imported/entered before COA Deployment as Segment Attributes appear only after Deployment.

Value Set - Validation Types

  1. Independent: Pre-defined list of Values (COA Value Sets should be of this type)
  2. Dependent: Pre-defined lists of values (Each list is based on one value of Independent Value Set)
  3. Table: Pre-defined list of values based on table(s) SQL
  4. Subset: Selected Values from an Independent Value Set
  5. Format Only: Any Value as per format

















# 2. COA Structure

Task: Manage Chart of Accounts Structures

  1. A Structure binds different Lists (E.g. Co, A/c, CC. Product) together, and each one is called a Segment

  2. Segment Label identifies the usage of a Segment (E.g. Primary Balancing Segment,Natural Account Segment,Cost Center Segment etc). This allows Oracle to build specific functionality for the segment.

  3. Segment Labels were referred as Flexfield Qualifiers in eBusiness Suite

  4. When we bind several Fields (Segments/Lists) together, it is called Flexfield.

  5. COA is a seeded Key flexfield called Accounting Flexfield.

















# 3. COA Structure Instance

Task: Manage Chart of Accounts Structure Instances

  1. A COA Structure Instance is one Instance of the Structure with some modifications. Likewise, multiple Instances can be created from a COA Structure

  2. Each COA Instance is considered as one COA. When multiple COA Instances are created from one Structure, most of the structure definition can be re-used

  3. Each COA Instance can have different Default Value

















# 4. Deployment

Task: Manage Chart of Accounts Structures

  1. Once COA Structure and COA Instance definition is complete, it can be Deployed for further use.
  2. Deployment validates all COA Structures and Instances and makes it available to use. It is similar to EBS GL Compile flexfield.
  3. Main difference (between Deployment and Compilation) is that Fusion GL COA Deployment considers ALL COAs whereas EBS GL Compilation considers only ONE COA
  4. Once Deployment is successful, the COA can be used for a Ledger

















# 5. Enter/Import Values

Tools:

  1. FBDI = File Based Data Import - Excel Spreadsheet
  2. RI = Rapid Implementation - Excel Spreadsheet

  1. COA Value Sets, Structures and Instances can be configured Manually or can be Imported using Rapid Implementation Spreadsheet Templates

  2. COA Values can be entered Manually or they can be Imported from Spreadsheets (FBDI, RI Templates)

  3. Along with Values, Account Hierarcy and Tree Labels can also be Imported for the Values (Hierarchy indicates Parent/Child relationships between values of the same segment) using FBDI/RI

















# 7. Single/Common/Global COA

  1. Same COA can be used by multiple Ledgers

  2. When a Group of companies uses same COA for all its Ledgers across multiple countries, it is called as Global COA

  3. Single/Common/Global COA, although a bit difficult to implement practically, gives many benefits by streamlining accounting process across multiple Ledgers (i.e. across multiple Countries, Businesses, Regions etc)
